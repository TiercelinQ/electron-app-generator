# /electron-run-tests ‚Äî Executable verification

## Role
QA engineer ‚Äî prove the project is clean and buildable with real commands.

## Goal
Run the verification ladder, surface any failure with its output, and confirm a clean state or hand off failures to `/electron-fix-issue`.

## Deliverable
A short French report stating what passed, with command output as proof; or a list of failures.

---

> If the project root has not been provided in this flow, first ask: `Racine du projet ? (chemin du dossier)`.

## Steps

1. **Apply `rules/verification.md ¬ßA`** in order:
   ```
   npm install                  # + postinstall (ensure-electron.cjs)
   npm run typecheck            # tsc --noEmit (node + web) ‚Äî expect clean
   npm run lint                 # eslint ‚Äî expect clean
   npm run build                # electron-vite build ‚Äî expect success
   npm run dist                 # only when packaging is requested
   ```
   - better-sqlite3 present ‚Üí `npx electron-builder install-app-deps` after install.
   - `Error: Electron uninstall` ‚Üí `npm run postinstall`.
2. **Capture output**. A non-zero exit or any reported error is a failure ‚Äî quote the relevant lines.
3. **Report** (French): which commands passed, with proof. On failure: name the command, the `file:line`, and hand off to `/electron-fix-issue` for the root cause (do not patch here unless the user asks).
4. If the **Node/npm environment is not available**: say so plainly, do the static read-through (`rules/verification.md ¬ßB`), and list the exact commands the user must run themselves. Never claim a clean typecheck you could not execute.

## Test creation policy

- Tests are opt-in at Phase 1 (Q6). If enabled, the suite (Vitest + Testing Library) was delivered in its dedicated batch ‚Äî run it (`npm test`).
- **Do not create a test suite if the project has none** (Q6 = No). The default verification is typecheck + lint + build.
- If the user **explicitly asks** to add tests later: follow `@rules/tests.md` (Vitest + `@testing-library/react` + jsdom, layer coverage, mocked model/DB and `window.api`). Confirm scope, then add the dev dependencies + `vitest.config.ts` + the `"test"` script.

## Anti-patterns ‚Äî what NOT to do
- **Do not** report "all green" without having actually run the commands (or without flagging that the env was unavailable).
- **Do not** fix code silently inside this skill ‚Äî verification reports; fixing is `/electron-fix-issue` (root cause) or `/electron-add-feature`.
- **Do not** scaffold tests or add a test runner the user did not ask for.
- **Do not** lower a check (skip typecheck, disable a lint rule) to make the report pass.

## When the user asks something adjacent
- **"Make the build pass"** ‚Üí if it fails, the root cause is in code/config; classify with `/electron-fix-issue`. Do not disable checks to force green.
- **"Add coverage"** ‚Üí that is explicit test creation; confirm scope + runner, then write tests.
